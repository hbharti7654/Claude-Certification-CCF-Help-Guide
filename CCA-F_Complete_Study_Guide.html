🎯 How to Eliminate Wrong Options & Find Correct Answers

The exam is about architectural judgment. There are consistent patterns in what's right and wrong. Master these and you'll eliminate 2 options immediately on most questions.

The Elimination Framework

  1. Identify the question type. Is it about termination/control flow? Prompt design? Error handling? Tool schema design? Each type has its own "right answer DNA."
  2. Eliminate anything that hard-codes arbitrary limits. "Stop after 10 iterations regardless of state" is always wrong. Robust systems respond to real signals, not capped counters.
  3. Eliminate anything that parses natural language for control flow. "Check if the assistant text contains the word 'done'" is always wrong. Use stop_reason, structured output, or explicit signals.
  4. Eliminate anything that bypasses structured validation. "Trust the model output directly without schema validation" is wrong for production-critical data.
  5. Eliminate answers that share context where it should be isolated. Subagents should not share full conversation history between each other unless explicitly needed.
  6. Pick the answer that uses the official/explicit mechanism Claude provides for that use case.

Distractor Patterns to Watch For

Distractor TypeExample (Wrong)Why It's WrongCorrect Pattern
Arbitrary cap "Limit agent to 10 tool calls max" Brittle — fails short, ignores completion signals Terminate on stop_reason === "end_turn"
Natural language parsing "Stop if assistant says 'completed'" Fragile string matching, not a proper contract Use structured stop signals or stop_reason
Self-review illusion "Ask Claude to review its own output in the same turn" Model retains generation context — can't catch its own errors Use a separate independent instance for review
Context leakage "Pass full orchestrator history to every subagent" Pollutes reasoning, wastes tokens, causes confusion Pass only task-relevant context to each subagent
Prompt-only guardrail "Tell the model in the prompt not to access X" Not enforceable — prompts are advisory Programmatic enforcement at the tool/infra layer
Overfat tools "Combine search + filter + summarize into one tool" Forces the model to make interior decisions it can't steer Separate tools for each logical operation
No nullable fields "Require all fields in extraction schema" Forces hallucination when data is absent Make optional fields nullable in JSON schema
🏆 Golden Rules for This Exam
  • The most robust, explicit, programmatic answer is almost always right
  • The most fragile, heuristic, or prompt-only answer is almost always wrong
  • Independent instances beat self-review, every time
  • Context isolation beats sharing everything, every time
  • When in doubt, ask: "Would this break in production if the model is slightly inconsistent?" If yes, eliminate it.
Domain 1 — 27%

🤖 Agentic Architecture & Orchestration

This is the biggest domain. Expect ~16 questions. Master this domain and you're already most of the way to passing.

What Is an Agentic System?

An agent is a system where Claude decides what to do next, rather than following a pre-scripted sequence. It has: a goal, access to tools, and a loop that runs until the goal is complete.

Agentic System
Claude autonomously decides tool usage and action sequence to complete a goal
Workflow
Pre-defined sequence of steps — Claude fills in each step, but the order is fixed
Conversational System
Back-and-forth chat — no persistent action loop, no tool execution autonomy
📌 When to use which
  • Agentic: tasks are complex, multi-step, dynamic, require tool calls across varying sequences
  • Workflow: steps are known upfront, order matters, each step can be pre-defined
  • Conversational: user is the decision-maker, Claude is advisor/assistant, no autonomy needed

The Agentic Loop — Core Architecture

This is tested heavily. Know every piece of this loop:

1. Send request to Claude API (with tools defined)
2. Receive response → check stop_reason
   - "tool_use" → Claude wants to call a tool
   - "end_turn" → Claude is done, loop terminates
3. If "tool_use" → execute the requested tool
4. Append tool result to conversation history
5. Send updated history back to Claude → go to step 2
🚫 Exam Traps in Agentic Loop
  • WRONG: Stop when assistant message contains "task complete" → natural language parsing, unreliable
  • WRONG: Set a fixed max iteration cap as the only stopping mechanism → arbitrary, brittle
  • WRONG: Check for absence of tool_use blocks to terminate → indirect, fragile
  • RIGHT: Terminate when stop_reason === "end_turn" — this is the official contract

Multi-Agent Topology Patterns

Hub-and-Spoke
One orchestrator agent coordinates multiple specialist subagents. Most common pattern. Orchestrator holds state.
Pipeline
Output of one agent feeds input of the next. Good for sequential transforms (extract → analyze → summarize).
Peer-to-Peer
Agents communicate directly. More complex. Used when cross-agent collaboration is dynamic.

Orchestrator vs Subagent Roles

AspectOrchestratorSubagent
ResponsibilityDecompose task, delegate, aggregate resultsExecute one focused task and return result
Context it holdsFull task state, user goal, all subagent outputsOnly what's needed for its specific task
Decision-makingWhich subagent to call next, when to stopHow to complete its assigned task
ScopeBroad, cross-cuttingNarrow, specialized
🔑 Context Isolation Rule

Subagents should receive only task-relevant context, not the orchestrator's full history. This prevents reasoning pollution, reduces token cost, and improves subagent accuracy.

Task Decomposition & Planning

  • Sequential execution: steps depend on each other → run one at a time. Use when step B needs step A's output.
  • Parallel execution: steps are independent → run simultaneously. Use to reduce latency when tasks don't share dependencies.
  • Dynamic replanning: the agent updates its plan based on intermediate results. Essential for open-ended tasks.
  • Ambiguity handling: when task is unclear, the agent should ask for clarification before acting, not guess.

Error Classification & Fallback

Tool Errors
The external service/API failed. Retry with backoff. After N failures, escalate or use fallback tool.
Reasoning Errors
Claude made a logic mistake. Use independent review instance. Don't retry the same prompt blindly.
Environment Errors
Infrastructure/state issue. Log, surface to human-in-the-loop, checkpoint state for resume.

Human-in-the-Loop (HITL) Design

Know WHEN to escalate to a human, not just how:

  • When the task involves irreversible actions (deleting data, sending emails, charging money)
  • When the agent's confidence is below a threshold you define
  • When the request is out of scope for the agent's authorized capabilities
  • When sensitive data or regulated actions are involved
  • Customer support scenario: escalate billing disputes over a certain threshold, legal threats, abusive interactions

Guardrails: Programmatic vs Prompt-Based

⚠️ This Is a High-Frequency Exam Trap

Prompt-based guardrails ("Tell Claude in the system prompt not to do X") are advisory only — they can be circumvented or ignored. For production safety, programmatic enforcement at the tool, API, or infrastructure layer is required.

Example: Don't say "only read files in /safe". Instead, enforce that at the filesystem/tool level so the tool physically cannot access /unsafe.

Multi-Pass Review Architecture

  • A model reviewing its own output in the same session is ineffective — it retains its original reasoning context and is unlikely to question its own decisions
  • Use a separate, independent Claude instance that doesn't have the generator's context
  • For large codebases, split review into local passes (per-file issues) + integration passes (cross-file data flow) to avoid attention dilution

State & Memory Types

In-Context Memory
Conversation history in the current context window. Fast, but temporary and size-limited.
External Memory
Database or vector store. Persistent, scalable. Must be explicitly retrieved and injected.
Domain 2 — 20%

💻 Claude Code Configuration & Workflows

~12 questions. Tests your knowledge of Claude Code's configuration system, tooling, and CI/CD usage.

What Is Claude Code?

Claude Code is a command-line AI coding assistant. It has its own tool system (Read, Write, Bash, Grep, Glob, WebFetch, etc.) and a configuration hierarchy via CLAUDE.md files.

CLAUDE.md Hierarchy — Know This Cold

Configuration cascades from broad to specific. More specific files override less specific ones.

LevelFile LocationScopeWho Controls It
User~/.claude/CLAUDE.mdAll projects for this userIndividual developer
Project (root)./CLAUDE.mdEntire projectTeam/repo owner
Subdirectory./subdir/CLAUDE.mdThat directory and belowModule owner
✅ Precedence Rule

More specific (deeper in the directory tree) overrides less specific. Subdirectory CLAUDE.md beats project root CLAUDE.md beats user CLAUDE.md.

.claudeignore

Works like .gitignore — tells Claude Code which files and directories to exclude from its context. Use to prevent Claude from reading secrets, build artifacts, or irrelevant large files.

Custom Slash Commands

  • Defined in .claude/commands/ directory as Markdown files
  • The filename becomes the command name: review-pr.md → /review-pr
  • The file content is the prompt template Claude executes
  • Can include argument placeholders for dynamic content
  • Distributed to teams by committing to the project repo

Skills (SKILL.md)

  • SKILL.md files encode reusable best practices that Claude should follow for a specific task type
  • Have frontmatter metadata (name, description, triggers)
  • Claude reads the relevant SKILL.md before attempting a task type that matches
  • Enterprise: skills can be deployed across all team members via shared config

Hooks — Lifecycle Events

Hooks let you inject custom logic at key points in Claude Code's execution lifecycle.

PreToolCall
Runs before Claude executes a tool. Use for safety checks, logging, permission gates.
PostToolCall
Runs after a tool call. Use for logging results, side effects, audit trails.
Exit Codes
Hook exit code 0 = allow, non-zero = block. This is programmatic enforcement.
🔑 Hooks = Programmatic Guardrails

If an exam question asks how to prevent Claude Code from doing something dangerous (deleting files, accessing prod DBs), the answer involves hooks with non-zero exit codes — NOT just instructing it in CLAUDE.md.

Subagents in Claude Code

  • Claude Code can spawn subagents for delegated tasks
  • Each subagent gets its own scoped context — they don't inherit the full parent session
  • Subagents are configured with specific tool permissions and constraints

Claude Code in CI/CD

  • -p "your prompt" — non-interactive (headless) mode for automation
  • --output-format json — structured output for parsing in pipelines
  • --no-interactive — prevents Claude from asking clarifying questions mid-run
  • Session isolation: each CI run should start a fresh session to prevent context leakage between runs
  • For code review in CI: minimize false positives by giving clear scope (what changed, what to ignore)

Plan Mode

Before Claude Code takes actions, it can enter plan mode where it explains what it intends to do and waits for confirmation. Use for high-risk, irreversible, or complex operations to give the human a chance to course-correct.

Writing Effective CLAUDE.md Instructions

  • Be specific and prescriptive, not vague ("always use TypeScript strict mode" not "write good code")
  • Define what tools Claude is not allowed to use in this context
  • Include project-specific conventions (testing frameworks, code style, branch naming)
  • Keep team-level rules in the project root CLAUDE.md, personal preferences in user-level
Domain 3 — 20%

✍️ Prompt Engineering & Structured Output

~12 questions. Tests both core prompting techniques and structured output reliability for production systems.

Core Prompt Engineering Principles

System Prompt
Sets role, constraints, output format, persona. Applies to the whole conversation. Put stable context here.
User Prompt
The specific task or question. Dynamic per request.
Few-Shot Examples
Showing Claude input→output pairs before the actual task. Dramatically improves format compliance.

The PRECISE Framework (Exam-tested)

LetterElementWhat It Means
PPersonaWho Claude should act as ("You are a senior security engineer…")
RRoleThe task role it should play
EExamplesFew-shot examples of good input/output
CConstraintsWhat it must or must not do
IInstructionsStep-by-step guidance for the task
SStyleTone, format, length requirements
EEvaluationHow success is measured

Few-Shot Prompting

  • Show 2-5 examples of what good output looks like before the actual request
  • Especially effective for format compliance — if you want JSON, show JSON examples
  • Examples should cover edge cases, not just the happy path
  • Negative examples (showing what NOT to do) can be as powerful as positive ones

Chain-of-Thought (CoT) Prompting

  • Ask Claude to "think step by step" or "reason through this before giving an answer"
  • Significantly improves accuracy on multi-step reasoning tasks
  • For production: consider prompting for CoT in a scratchpad, then extract the final answer separately
  • Extended thinking: Claude can be configured to produce longer internal reasoning chains

Structured Output — JSON Schema Enforcement

This is critical for the data extraction scenario and any production use case that parses Claude's output programmatically.

{
  "type": "object",
  "required": ["invoice_id", "total"],
  "properties": {
    "invoice_id": { "type": "string" },
    "total": { "type": "number" },
    "line_items": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "description": { "type": "string" },
          "amount": { "type": ["number", "null"] }  // ← nullable!
        }
      }
    }
  }
}
⚠️ Nullable Fields — High-Exam-Value Concept

When extracting data from documents, not all fields will always be present. If you make every field required, Claude will hallucinate values to fill them. Make optional fields nullable ("type": ["string", "null"]) so Claude can return null instead of inventing data.

Validation Retry Loop

Don't trust unvalidated structured output in production. Use this pattern:

1. Send prompt requesting JSON
2. Receive output
3. Validate against JSON schema
4. If validation FAILS → send error message + retry prompt to Claude
5. Repeat up to N times
6. If still failing after N retries → escalate / fallback

Prompt Injection Defense

  • Prompt injection: malicious content in user input or tool results that tries to override system instructions
  • Defense: clearly delimit user content from instructions (use XML tags, explicit markers)
  • Defense: don't blindly execute instructions found in tool results without validation
  • Defense: programmatic filters on what tool results can be passed back to Claude

Prompting for Code Generation

  • Include the language, framework, and version in the system prompt
  • Specify testing requirements upfront
  • Ask for code + explanation in separate sections
  • Use role prompting: "You are a senior Python engineer who writes production-quality, well-tested code"

Evaluating Prompt Quality

  • Define clear success metrics before writing the prompt
  • Test with adversarial inputs — edge cases, ambiguous phrasing, missing data
  • Use automated evaluation with expected outputs when possible
  • Track false positive rate for tasks like code review — high FPR destroys developer trust
Domain 4 — 18%

🔧 Tool Design & MCP Integration

~11 questions. Tests tool schema design principles and Model Context Protocol (MCP) architecture.

What Is MCP (Model Context Protocol)?

MCP is Anthropic's open standard for connecting Claude to external tools and data sources. Think of it as the "USB-C for AI" — a universal protocol that any service can implement to become usable by Claude.

MCP Server
A service that exposes tools/resources Claude can call. You build this for your external service.
MCP Client
Claude (or the host application) that calls the MCP server's tools.
Transport Layers
stdio: local process communication. SSE (Server-Sent Events): remote/HTTP-based communication.

MCP Transport: stdio vs SSE

TransportUse WhenKey Characteristic
stdioLocal tools running on the same machine as Claude CodeLow latency, simple, no network config needed
SSERemote services, cloud APIs, multi-user scenariosHTTP-based, works across network boundaries

Tool Schema Design Principles

Tool schemas define what Claude can call. Good schemas make Claude more accurate. Poor schemas cause reasoning errors.

🔑 Core Tool Design Rules
  • One logical operation per tool. Don't combine search + filter + summarize into one tool. Give Claude individual, composable tools.
  • Clear, descriptive names. search_customer_by_email not customerSearch.
  • Rich descriptions. Describe exactly when to use this tool vs alternatives. Claude reads descriptions to decide which tool to call.
  • Required vs optional parameters. Only make things required if they truly are.
  • Enum constraints. When a parameter has a fixed set of valid values, enumerate them. This prevents the model from guessing wrong values.

Tool Schema Example (Right vs Wrong)

AspectWrongRight
Tool namedoStuffupdate_ticket_status
Description"Updates things""Updates the status of a support ticket. Use when the user's issue has been resolved or requires escalation."
Status param{"type": "string"}{"type": "string", "enum": ["open", "resolved", "escalated"]}
GranularityOne tool handles search, update, and deleteSeparate tools: search_ticket, update_ticket, delete_ticket

Preventing Reasoning Overload

If Claude has too many tools, or tools that are too complex/overlapping, it struggles to choose the right one. Mitigation:

  • Limit tool count per agent to what's relevant for its task scope
  • Use different tool sets for different subagents (each subagent only sees its own tools)
  • Group related tools under a clear domain (all customer tools vs all billing tools)

MCP Security Considerations

  • Authentication: MCP servers should verify that the calling client is authorized
  • Scope limiting: each MCP server should only expose the minimum tools needed for its use case
  • Tool result validation: validate what the MCP server returns before passing it back to Claude (defense against injection)
  • Read vs write separation: separate read-only tools from write/mutation tools; give agents only what they need

Domain 5 — 15%

🧠 Context Management & Reliability

~9 questions. Tests how to manage context windows, avoid degradation, and build reliable long-running systems.

Context Window Fundamentals

  • Context window = the amount of text Claude can "see" at once (input + output tokens combined)
  • As context fills up, performance can degrade — Claude may lose track of earlier content
  • Attention dilution: when too much is in context, Claude may not give equal weight to all of it
  • For large codebases or long conversations, you must manage context actively

Context Management Strategies

Summarization
Compress earlier conversation turns into a summary. Keeps context fresh at the cost of some detail loss.
Retrieval-Augmented
Store information externally (vector DB), retrieve only what's relevant for the current turn and inject it.
Sliding Window
Keep only the last N turns in context. Simple but risks losing important earlier context.
Prompt Caching
Cache expensive prefix content (system prompt, large doc) to reduce cost and latency on repeated calls.

Prompt Caching

  • Expensive stable content (system prompts, reference documents) can be cached at the API level
  • Cached tokens are cheaper to process on subsequent calls
  • Cache the parts of the prompt that don't change between requests
  • Dynamic content (the user's current question) is NOT cached

Long Context Reliability Patterns

  • Chunking: break large documents into chunks, process separately, then aggregate
  • Map-reduce pattern: map = process each chunk independently; reduce = combine results into final answer
  • Per-file + integration passes for code review (see Domain 1 — multi-pass review)

Handoff Patterns Between Sessions

Claude has no memory between separate API calls. For workflows that span sessions:

  • Save a structured state object to external storage at checkpoints
  • On new session start, inject the state object into the system prompt or first user message
  • Include enough context to resume meaningfully, not the entire history verbatim

Confidence Calibration

  • Ask Claude to self-report confidence alongside each output: "Rate your confidence 1-10 and explain why"
  • Route low-confidence outputs to a human reviewer or secondary verification step
  • Don't use a single arbitrary threshold for all cases — calibrate per task type
  • For verification passes: the model self-reports confidence per finding, enabling calibrated review routing

Reliability in Production

  • Retry with exponential backoff for transient errors (rate limits, 5xx errors)
  • Fallback models: if primary model fails or is too slow, route to a backup
  • Timeout handling: set explicit timeouts; never let an agent loop forever on a stuck state
  • Idempotent tool calls: design tools so calling them twice doesn't double the effect (important for retries)
  • Structured logging: log every tool call, its inputs, outputs, and the stop_reason. Essential for debugging agentic loops.

Context Window Anti-Patterns

⚠️ Common Context Mistakes (Exam Traps)
  • Dumping entire codebase into context → attention dilution, worse results than targeted retrieval
  • Including all past conversation turns verbatim → context explosion, costly, eventually fails
  • Passing full orchestrator history to subagents → pollution, confusion, wasted tokens
  • Never clearing context between CI runs → previous run artifacts affect current run
